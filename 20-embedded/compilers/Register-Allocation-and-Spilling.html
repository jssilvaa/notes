Register Allocation and Spilling

Register Allocation and Spilling

May 16, 20265 min read

  • embedded/compilers
  • math/optimization

Summary

After instruction selection, the compiler must map an unbounded set of SSA virtual registers to a fixed physical register file plus stack. The canonical formulation is graph k-coloring on the interference graph, which is NP-complete for k≥3, so production compilers use heuristic allocators (Chaitin-Briggs, linear scan, greedy, PBQP). When coloring fails, the allocator picks a vreg to spill to memory according to a cost heuristic and retries.

Core Mechanism

Setup. Liveness analysis produces a live range [sv​,ev​] for each vreg v. Two vregs interfere if their ranges overlap. The interference graph G=(V,E) has V=vregs and E=(u,v):ranges overlap. With k physical registers, the allocation problem is:

minS⊆V​∑v∈S​c(v)s.t. G[V∖S] is k-colorable

where c(v) is the spill cost of v and S is the spill set. Both the feasibility check (k-colorability) and the objective are NP-hard, so all practical work approximates simultaneously.

Chaitin-Briggs (graph coloring). Iterate until colored:

  1. Simplify: while ∃v with deg(v)<k, push v onto stack, remove from G. The simplification lemma: a node with fewer than k neighbors is always colorable after its neighbors are.
  2. Potential spill: if no such v exists, pick spill candidate v∗=argminv​c(v), remove it.
  3. Select: pop stack, greedily assign a color ∈/ colors of colored neighbors.
  4. Rewrite: if any node was marked for spill, insert loads/stores at uses/defs, re-run liveness, restart.

Chaitin’s spill cost heuristic:

c(v)=degG​(v)(∣defs(v)∣+∣uses(v)∣)⋅wloop​(v)​

High degree → spilling relieves more pressure; low use count → cheap to reload.

Linear scan. Sort live intervals by start point. Maintain active set A of currently-allocated intervals. For each new interval i: expire any j∈A with ej​<si​ (free its reg); if ∣A∣<k assign a free reg; else spill the interval in A∪i with latest end point. O(nlogn) vs. Chaitin’s O(n2), at the cost of over-approximating interference (two intervals that overlap on the timeline may not actually interfere in the SSA dataflow sense).

Iterated Register Coalescing (George-Appel). Extends Chaitin-Briggs with conservative move elimination. Coalesce a,b across mov a, b iff one of:

  • George: every neighbor t of a with deg(t)≥k already interferes with b.
  • Briggs: the merged node ab has fewer than k neighbors of degree ≥k.

These tests guarantee coalescing never makes a colorable graph uncolorable.

PBQP (Partitioned Boolean Quadratic Problem). Formulates allocation as minimizing ∑v​Cv​(xv​)+∑(u,v)∈E​Cuv​(xu​,xv​) where xv​ is v‘s register assignment, Cv​ is a per-node cost vector (preferences, spill cost), and Cuv​ is an interference penalty matrix. Solved via belief propagation. Still NP-hard but tractable for realistic graph sizes.

Why It Matters

This is the pass that decides, for every variable you write, whether it lives in a register or thrashes through L1 — and it runs every time anything is compiled. Concretely:

  • JIT compilers (V8, HotSpot, LLVM -O0) use linear scan because compile time is on the critical path of execution; the spill penalty is acceptable.
  • AOT optimizing builds (-O2/-O3) use greedy or IRC because the user pays compile time once and runtime forever.
  • GPU compilers treat register allocation as an occupancy problem: a kernel using too many registers reduces the number of concurrent warps, so the spill/occupancy tradeoff is more important than spill count alone.
  • Embedded targets with k=8 or fewer GPRs (AVR, MSP430) make spill decisions visible in benchmark numbers — a missed coalesce in a tight loop can double cycle count.

Limitations

  • All practical allocators are heuristic; for adversarial graphs they spill more than the optimum. The gap rarely matters but is unbounded in principle.
  • Spill cost models treat all memory accesses identically — they don’t model L1 vs. L2 reload, store-to-load forwarding, or the fact that a spilled vreg in a hot loop may be the actual bottleneck even with low “cost.”
  • Linear scan’s interval abstraction is strictly weaker than the interference graph: it can force spills that graph coloring would avoid. The damage is small on straight-line code, larger on irregular control flow.
  • Phase ordering: register allocation and instruction scheduling each constrain the other. Schedule-first inflates live ranges; allocate-first removes scheduling freedom. Most compilers pick one and live with it.
  • Aggressive coalescing without George/Briggs tests increases degree and causes spills (the original Chaitin bug).
  • SSA destruction (where the allocator leaves SSA form) introduces parallel copies that interact subtly with coalescing.

Open Questions

  • How much do learned spill-cost predictors (cf. MLGO-style approaches) actually move the needle on standard benchmarks vs. the Chaitin heuristic? The function looks crude but is hard to beat.
  • For GPU codegen, is there a principled way to fold occupancy targets into the allocator’s objective rather than running allocation under a hard register budget set by an outer loop?
  • PBQP is “real” optimization but rarely shipped — what’s the actual compile-time cost on a kernel-scale function, and where does it win over greedy in practice?
  • Live-range splitting (breaking a long-lived vreg into pieces, each separately allocatable) is where modern allocators get most of their wins over Chaitin-Briggs. Implementing it well is the actual engineering problem.
